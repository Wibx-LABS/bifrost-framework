# bifrost-graphify-ref

This skill is the protocol for **lookup before invention**. Bifrost agents (especially `@CodeGen` and `@Intake`) will produce silently-wrong code if they invent endpoints / components / patterns instead of using the ones that already exist. The Wiboo monorepo has a populated knowledge layer; the agent's job is to consult it first, every time.

The skill works in two phases. **Today**: `graph.json` (the structured architecture graph generated by Graphify) does not yet exist. The knowledge layer is the five markdown files in `projects/bifrost-framework/knowledge/`, plus the actual TypeScript code at `libs/commonlib/src/lib/constants/api.ts` (the URL factory) and `apps/<app>/src/app/core/api/*.api.ts` (per-app HTTP service classes). **Future** (post-`GRAPHIFY_SEEDING_PLAN.md` execution): `graph.json` is the structured index, and the markdown files become semantic backstops. Both states are covered below.

## The questions this skill answers

When you, as an agent, find yourself about to invent something, route through this skill first. Common questions:

| Question | Where the answer lives |
|---|---|
| What endpoints exist for search / users / wallet / payments / etc.? | `knowledge/API_CONTRACTS.md` (when seeded) → `libs/commonlib/src/lib/constants/api.ts` (today) → per-app `*.api.ts` |
| Is there an existing `app-<thing>` component? | `knowledge/COMPONENT_LIBRARY.md` |
| What naming convention applies to <thing>? | `knowledge/NAMING_CONVENTIONS.md` |
| What's the version of <library>? | `knowledge/TECH_STACK.md` |
| What gotchas / known issues exist in <area>? | `knowledge/GOTCHAS.md` |
| What's the file structure of the apps / libs? | `knowledge/FRONTEND_REPOSITORY_MANUAL.md` §2 + §5 + §6 |
| What state-management pattern does Wiboo use? | `knowledge/FRONTEND_REPOSITORY_MANUAL.md` §7 + `bifrost-state-management` skill |
| What's Wiboo's i18n setup? | `knowledge/FRONTEND_REPOSITORY_MANUAL.md` §9 |
| What ESLint rules are enforced? | `knowledge/TECH_STACK.md` ESLint Rules Summary + `bifrost-code-standards` skill |
| What Brazilian locale gotchas apply (CPF, CNPJ, phone, currency)? | `knowledge/GOTCHAS.md` §Brazilian Locale Gotchas |

If the answer to your question is in the knowledge layer, you must use what's there. Don't invent.

## State A — today (no `graph.json`)

### The knowledge files (start here)

```
projects/bifrost-framework/knowledge/
├── FRONTEND_REPOSITORY_MANUAL.md   # ~60KB — the deep reference (apps, libs, NgRx, styling, i18n, …)
├── API_CONTRACTS.md                # ⚠️ NOT YET SEEDED — Backend-owned, pending. When created: endpoint signatures.
├── COMPONENT_LIBRARY.md            # ~50+ commonlib components (form, container, display, specialized, layout)
├── NAMING_CONVENTIONS.md           # Naming + ESLint rules
├── TECH_STACK.md                   # Versions, configuration, performance targets
└── GOTCHAS.md                      # Known issues, security rules, common mistakes
```

**`API_CONTRACTS.md` is referenced by the framework spec but does not yet exist.** Until it's seeded (per `GRAPHIFY_SEEDING_PLAN.md` in the framework's docs), the working source of truth for "what endpoints exist?" is the actual TypeScript code below.

### The code-side knowledge (until `API_CONTRACTS.md` ships)

| Question | File |
|---|---|
| What endpoints exist (definitive)? | `libs/commonlib/src/lib/constants/api.ts` — the central URL factory, organized by domain. |
| What request/response shapes are used? | `libs/commonlib/src/lib/services/<domain>.service.ts` (or per-app `core/api/*.api.ts`) — the wrapper services that consume `api.*`. The TypeScript types on those service methods ARE the contract. |
| What DTO types match the wire format? | `libs/commonlib/src/lib/models/*.dto.ts` and per-app equivalents. |
| What model types are used in the app? | `libs/commonlib/src/lib/models/*.model.ts` and per-app equivalents. |
| What adapters bridge DTO → model? | `libs/commonlib/src/lib/adapters/*.adapter.ts`. |

**The lookup pattern, today:**

```
Agent question: "Does an endpoint exist for marking a notification as read?"
Step 1: Open libs/commonlib/src/lib/constants/api.ts.
Step 2: Search for `notification` in the file (or grep at the repo level).
Step 3: If found (e.g., `api.notifications.markRead(id)`): use it. Done.
Step 4: If not found: open libs/commonlib/src/lib/services/ to check whether a service wraps a related endpoint.
Step 5: If still not found: this is a real gap. Two options:
        (a) The endpoint exists but isn't in api.ts → flag in IMPACT.md and ask Backend to confirm + add.
        (b) The endpoint doesn't exist yet → this is a Backend dependency. List in TRAJECTORY §2 (blocking dependencies).
        DO NOT invent the URL inline.
```

Same pattern for components (`knowledge/COMPONENT_LIBRARY.md` → `libs/commonlib/src/lib/components/`).

Same pattern for naming/style (`knowledge/NAMING_CONVENTIONS.md` is canonical; ESLint config is the enforced subset).

### Cross-cutting search — when you don't know which file

If you genuinely don't know which knowledge file has the answer, search the whole knowledge layer:

```bash
grep -rn -i "<keyword>" projects/bifrost-framework/knowledge/
```

Or, if the question is about code:

```bash
grep -rn "<keyword>" libs/commonlib/ apps/<app>/src/app/core/
```

Read the surrounding context. The cost of one extra read is negligible; the cost of inventing a pattern that already has a different established form is rework + Backend frustration.

### The `*.api.ts` files

Each app has a `core/api/` folder with HTTP service classes — these wrap the `api.*` URL factory and add typing. Examples (illustrative, exact paths vary):

```
apps/business/src/app/core/api/
├── searching.api.ts        # SearchingApiService — wraps api.searching.*
├── transactions.api.ts     # TransactionsApiService — wraps api.transactions.*
└── ...
```

When you (`@CodeGen`) need to call an endpoint, look here first to find an existing service. If a service already exists for the domain, use it (inject it, call its method); don't create a parallel service that calls the same endpoints with slightly different shapes.

## State B — future (`graph.json` is seeded)

When the Backend team executes `GRAPHIFY_SEEDING_PLAN.md`, the result is a `graph.json` (and a companion `GRAPH_REPORT.md`) under `knowledge/`. The graph carries:

- **Component inventory** — every `@Component` class, its selector, file path, `@Input`/`@Output` signatures, dependencies.
- **Service inventory** — every `@Injectable`, methods, return types, dependencies.
- **State management graph** — actions / reducers / selectors / effects, their cross-references.
- **API usage graph** — which components/services call which endpoints.
- **Type definitions** — every interface / type / enum, where defined, where used.
- **Dependency graph** — internal imports + external package usage.

When `graph.json` exists, queries become structured rather than grep-based. The agent reads `graph.json` directly (or via a helper utility provided by the framework) and asks:

```
graph.find('component', { selector: 'app-search-portal' })
  → { class: 'SearchPortalComponent', file: 'apps/business/src/app/containers/search-portal/...', inputs: [...], outputs: [...] }

graph.find('endpoint', { domain: 'searching' })
  → [ { name: 'queryPortals', method: 'POST', path: '/api/search/...', request: ..., response: ... }, ... ]

graph.find('action', { name: 'loadUser' })
  → { type: '[Profile Store] Load the user', payload: { id: 'string' }, fired_in: [...], handled_in: [...] }
```

**Until `graph.json` exists**, simulate the graph with grep + targeted file reads. The semantic answers (the markdown files) are stable; the structured index is what's missing.

When `graph.json` arrives, the workflow shifts to: query graph first (cheap, structured), fall back to markdown (semantic, narrative) only when the graph doesn't have the level of detail needed.

## How `@Intake` uses this skill (gap detection)

Per `bifrost-hr` and ADR-009, `@Intake` runs gap detection during `/bifrost:start`. This skill is what `@Intake` consults to evaluate the question "do my loaded skills + the knowledge layer cover this feature?".

The protocol:

1. Read PATIENT.md.
2. Identify the domains the feature touches (auth, payments, search, notifications, wallet, charts, …).
3. For each domain, ask:
   - Is there an existing `bifrost-<domain>` skill? (See the `bifrost-system-context` agent×skill matrix.)
   - Are there existing endpoints? (Query `libs/commonlib/src/lib/constants/api.ts`.)
   - Are there existing components? (Query `knowledge/COMPONENT_LIBRARY.md`.)
   - Are there relevant gotchas? (Query `knowledge/GOTCHAS.md`.)
4. If any domain is uncovered (no skill, no patterns, no gotchas), `@Intake` invokes `bifrost-hr` to bootstrap a new skill before locking TRAJECTORY.md.
5. If all domains are covered, `@Intake` proceeds to lock TRAJECTORY.md normally.

This skill is the *evidence collection* phase of the gap-detection protocol. Don't shortcut it — a missed gap at intake is a mid-flight Hard Stop later, costing more.

## How `@CodeGen` uses this skill (lookup-before-invention)

Every time `@CodeGen` is about to write a line that touches:

- An HTTP call → consult `bifrost-api-integration` for the pattern + this skill for the endpoint.
- A UI component → consult `bifrost-component-gen` for the pattern + this skill for whether the component exists.
- A state slice → consult `bifrost-state-management` for the pattern + this skill for whether the slice or actions already exist.
- A naming choice → consult `bifrost-code-standards` + this skill for the precise rule.

The pattern in pseudocode:

```
function generateCode(task) {
    consultLoadedSkillsForPattern(task);
    queryKnowledgeLayerForExistingArtifacts(task);   // ← this skill

    if (existingArtifactCovers(task)) {
        useExistingArtifact();
    } else {
        generateNew(/* following the loaded-skill pattern */);
    }
}
```

Skipping the `queryKnowledgeLayer` step is how silent context loss happens. Don't skip it.

## Self-review checklist

When `@CodeGen` finishes and walks `bifrost-code-review`:

- [ ] Every endpoint called exists in `api.<domain>.<endpoint>()`. (Confirmed by reading `libs/commonlib/src/lib/constants/api.ts`.)
- [ ] Every component used (other than newly-created feature-specific ones) is in `knowledge/COMPONENT_LIBRARY.md`. (Confirmed by name match.)
- [ ] Naming choices match `knowledge/NAMING_CONVENTIONS.md`. (Confirmed by ESLint pass.)
- [ ] Library versions match `knowledge/TECH_STACK.md`. (Confirmed by `package.json` not being modified, OR by an ADR if a version bump was authorized.)
- [ ] Known gotchas in the relevant area (`knowledge/GOTCHAS.md`) are not re-introduced.
- [ ] Where a new endpoint or new component was created, the addition is named in `IMPACT.md` and visible in HANDOFF.md so Backend can review.

When `@Intake` finishes:

- [ ] Every domain mentioned in PATIENT.md was checked against the knowledge layer.
- [ ] Domains without coverage triggered `bifrost-hr` (per ADR-009).
- [ ] TRAJECTORY.md §5 (External context) lists the prior incidents from `knowledge/GOTCHAS.md` that this feature must not re-introduce.
